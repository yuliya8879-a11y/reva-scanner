# Quality Loop для AI-разборов EyeScanner

## Цель

Сделать качество разборов управляемым: измеряемым, воспроизводимым и улучшаемым итерациями, а не «по ощущению».

## Где используется

- Ручной разбор первых запусков после изменений промптов (`app/services/algorithm.py`, `app/services/full_scan_ai_service.py`)
- Регулярная выборка готовых разборов из БД (по `scan.report`)
- Продуктовая аналитика (конверсия в следующий шаг, возвраты, обращения в поддержку)

## Рубрика оценки (0-2 балла за критерий)

Оценка: `0 = нет`, `1 = частично`, `2 = хорошо`.

### 1) Персонализация
- Есть упоминание конкретных вводных пользователя (имя, запрос, контекст анкеты)?
- Выводы связаны с данными, а не общими фразами?

### 2) Конкретика диагноза
- Названы конкретные блоки/узкие места без «воды»?
- Есть причинно-следственная связка, а не набор советов?

### 3) Практическая применимость
- Есть чёткие шаги, которые можно сделать сегодня/на неделе?
- Шаги наблюдаемы и проверяемы пользователем?

### 4) Тон и соответствие методу
- Текст соответствует стилю метода (прямой, без запугивания)?
- Нет противоречий с METHOD.md и актуальными алгоритм-блоками?

### 5) Безопасность и честность
- При недостатке данных явно сказано «недостаточно данных», без выдумок?
- Нет потенциально вредных/категоричных формулировок?

### 6) Структурная целостность
- Все обязательные блоки присутствуют (`архитектура`, `слепые_зоны`, `энергетические_блоки`, `команда`, `деньги`, `рекомендации`)?
- Нет сломанного формата или пустых блоков?

## Формула качества

- `quality_score = sum(критерии 1..6)` (максимум 12)
- Пороговые зоны:
  - `10-12`: стабильное качество
  - `7-9`: приемлемо, но нужны точечные правки
  - `<7`: обязательная доработка промпта/логики

## Операционный процесс

1. После любого изменения алгоритма/промпта выбрать 10-20 последних разборов.
2. Проставить баллы по рубрике для каждого разбора.
3. Зафиксировать 3 самых частых дефекта (например: «мало конкретики», «слабая персонализация»).
4. Сделать одну целевую правку промпта/форматирования.
5. Повторить замер на новой выборке.

## Продуктовые метрики (еженедельно)

- Конверсия `mini -> paid`
- Конверсия `paid -> complete report delivered`
- Доля пользователей, которые задают уточняющий вопрос после отчёта
- Доля возвратов/жалоб
- Средний `quality_score` по ручной выборке

## Критерии готовности релиза алгоритма

Изменение алгоритма считается «готовым к прод» только если:

1. Средний `quality_score >= 9` на выборке минимум 10 разборов.
2. Нет критических провалов по критерию «безопасность и честность».
3. Не ухудшились ключевые продуктовые метрики против предыдущей итерации.
